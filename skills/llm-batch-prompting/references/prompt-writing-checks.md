# 批量 Prompt 写作与自查规范

## Prompt 写作重点

批量任务的 prompt 要优先保证稳定性、一致性和可检查性。不要把重点放在调用方式或请求文件格式上；skill 只负责把 prompt 本身写清楚，并给出上线前检查方法。

## 必备结构

一个合格的 XML Prompt Contract 至少包含：

- `<background>`：业务背景、使用场景、结果消费方。
- `<role>`：模型角色和判断风格。
- `<task>`：单条输入要完成的任务。
- `<input_schema>`：输入字段名、类型、含义、是否必填。
- `<requirements>`：硬约束、禁止事项、判断原则。
- `<output_schema>`：输出字段、类型、枚举值、为空策略。
- `<examples>`：3-5 个贴近真实分布的示例。
- `<edge_cases>`：空值、冲突、越界、低置信度、无法判断的处理方式。
- `<current_input>`：当前单条输入。

## 写作规则

- 单条输入原则：prompt 中只处理一个 `<current_input>`，不要让模型在一次调用里处理多条业务数据。
- 先 schema 后任务：先明确输出字段，再组织任务说明和示例。
- 明确兜底策略：无法判断时返回什么、置信度如何设置、是否需要人工复核。
- 禁止隐含规则：所有业务判断标准都写进 `<requirements>`、`<label_definitions>` 或 `<decision_rules>`。
- 示例覆盖边界：不要只给标准正例，必须至少包含一个信息不足或冲突样例。
- 输出可解析：明确要求只输出合法 JSON，不输出 Markdown、代码块、解释性前后缀。
- ID 可回填：输入 schema 中应包含唯一 ID，并要求输出中原样返回。
- 版本可追踪：在 `<prompt_contract version="...">` 中记录 prompt 版本。

## Prompt 版本管理

每次修改 prompt 后都要提升版本号。小改措辞用 patch 版本，改判断规则用 minor 版本，改输出 schema 用 major 版本。

示例：

- `feedback-classifier-v1.0.0`
- `feedback-classifier-v1.1.0`
- `feedback-classifier-v2.0.0`

版本号不是模型或调用版本，而是 prompt contract 版本，用于对比效果和回溯问题。

## 自查问题

交付 prompt 前，逐项检查：

- 这个 prompt 是否不依赖对话历史？
- 用户把任意一条输入替换进 `<current_input>` 后，模型是否知道要做什么？
- 输出 schema 是否足够精确到字段类型、枚举值和空值策略？
- 所有标签定义是否互斥或有优先级？
- 示例是否覆盖正例、反例和边界例？
- 信息不足时是否有明确处理方式？
- 是否明确禁止额外解释、Markdown 和代码块？
- 是否能用脚本或人工快速判断输出是否合格？

## 常见风险

- 背景很长，但任务目标不清楚。
- 只写“请分类”，没有标签定义和优先级。
- 输出要求写成自然语言，但没有 schema。
- 示例和真实输入分布差异太大。
- 缺少低置信度和人工复核策略。
- prompt 中混合多条输入，模型输出漏项或错位。
- 改过 prompt 但没有版本号，导致评测结果不可追踪。
